→ How data flows through the system

Every Phase 1 spec is one box in this pipeline. The interfaces between boxes — the graph JSON, the rule YAML, the Assessment object — are fixed contracts, so each box can be built independently against its spec.

spec 005

Report Renderer

Markdown / HTML / JSON — identical content, byte-deterministic

spec 003

Rules (YAML) — feeds into the Evaluation Engine above

Declarative match + assert conditions, each with framework mappings (SOC 2, ISO 27001, PCI-DSS…). ~30 built-in rules ship in Phase 1.

spec 006

CLI (archscan) — orchestrates every box above

The free distribution wedge. archscan scan infra/ ties parser → rules → engine → report into one command, with a stable exit-code contract (0/1/2/3) for CI gating.

📚 Document map — four kinds of document

Each document answers a different question. Pick the one that matches what you need right now.

Why

PRD.md

Problem, personas, user stories, success metrics, explicit non-goals. Read this to understand who this is for and why Phase 1 is scoped the way it is.

What — core

specs 000–006

Implementation specs: graph schema, Terraform parser, rule format, evaluation engine, report renderer, CLI. Each has data structures, edge cases, and numbered acceptance tests.

What — quality bar

specs 007–011

Non-functional requirements, threat model, test strategy, lifecycle/versioning, rule-pack governance. This is what separates a prototype from something production-grade.

Proof

TRACEABILITY.md

Every requirement (R-01…R-12) traced to the spec that designs it and the test that proves it. A requirement with no test listed isn't done.

Why not X

decisions/ (8 ADRs)

Architecture Decision Records — the alternatives considered and rejected for HCL parsing, rule format, verdict logic, CLI stack, determinism, licensing, and scanner strategy.

🛣 Roadmap

Phase 1 is fully specced. Everything after it is sequenced but not yet written.

0

Validate in progress

10–15 customer interviews + a Terraform→graph spike, before writing a line of product code.

1

Core: graph, parser, engine, CLI fully specced

Free local CLI. Terraform → graph → SOC 2-mapped rule pack → scored report. Specs 000–011 cover this completely, including NFRs, threat model, and test strategy.

2

Custom guardrails planned

Org-authored rules, CI enforcement (GitHub Action), SARIF importer for external scanners (ADR-0008).

3

Hosted platform planned

Web dashboard, document ingestion, per-framework readiness views, evidence export, ISO 27001 pack. Commercial boundary per ADR-0007.

4

Modeling DSL + doc generation planned

Manual architecture declaration; living C4-style documentation generated from the graph; PCI-DSS pack.

5

Cloud scanning + drift planned

Read-only AWS connector; intended-vs-actual drift detection; HIPAA/GDPR technical-subset packs.

6

Design assistant planned

Propose guardrail-compliant designs from requirements — the only phase where an LLM sits inside a generative (not extractive) role.

✅ What makes this "production-grade" rather than a prototype spec

The gap between a working prototype and a production-grade one is usually invisible until it's expensive. These are the specific things this suite pins down that a minimal spec set would skip.

ConcernWhere it's addressedWhat it forces
Performance budgetsspec 007 §1Numeric targets (e.g. 5,000 resources in <30s, <512MB) — not "should be fast"
Security threatsspec 00810 named threats (YAML code exec, ReDoS, path traversal, XSS…) each with a required mitigation and a test
DeterminismADR-0006, specs 000/001/004/005Byte-identical output enforced by a dedicated CI job, not just "sorted where convenient"
Test rigorspec 009100% branch coverage on verdict logic, property-based tests, fuzz suite, real-world corpus job
Compliance mapping integrityspec 011Human SME sign-off required before any framework mapping ships — wrong mappings are worse than none
Versioning contractspec 010 §1Named "stable contracts" (exit codes, schema versions) that can never break without a major bump
Accessibilityspec 007 §5WCAG-aligned contrast, no color-only severity signaling, semantic HTML
Supply chainspec 008 T7/T8, spec 010 §4Zero-CVE dependency gate, PyPI trusted publishing (OIDC), 2FA from day one
Requirement→test traceabilityTRACEABILITY.mdEvery PRD requirement has a named test or is explicitly flagged as an accepted manual gap
What's still missing before this is truly "complete": nothing in the document set is implemented yet. Specs de-risk design mistakes; they don't replace writing the code, running it against real-world Terraform repos, and finding the gaps only execution reveals (spec 009 §6's corpus suite exists precisely for this reason).
