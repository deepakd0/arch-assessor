# Spec 006 ‚Äî CLI (`archscan`)

**Depends on:** specs 000‚Äì005. This is the integration layer and the free distribution wedge:
an engineer must get a useful assessment within five minutes of `pip install`.

## 1. Purpose

A single command-line tool wiring parser ‚Üí rules ‚Üí engine ‚Üí report, with CI-friendly exit
codes. Phase 2 turns this into a GitHub Action; the interface below is designed so that
requires no breaking changes.

## 2. Scope

**In scope:** `scan`, `graph`, `rules` subcommands; flag handling; exit codes; stderr/stdout
discipline. **Non-goals:** config files (`archscan.toml` ‚Äî Phase 2); watch mode; network
anything. The CLI must work fully offline.

## 3. Commands

Use `argparse` (stdlib ‚Äî no click/typer dependency). Program name `archscan`, wired via
`[project.scripts]` in `pyproject.toml` to `archassessor.cli.main:main`.

### 3.1 `archscan scan PATH [options]`

The main event: parse Terraform at `PATH`, evaluate rules, write a report.

| Flag | Default | Meaning |
|------|---------|---------|
| `--format {md,html,json}` | `md` | report format |
| `--output FILE` / `-o` | stdout | write report to file (UTF-8) |
| `--rules PATH` (repeatable) | ‚Äî | additional rule files/dirs (spec 003 loader) |
| `--no-builtin` | off | skip the built-in pack (use only `--rules`) |
| `--framework KEY` (repeatable) | all | limit the compliance-readiness section to these frameworks (`soc2`, `iso27001`, `pci_dss`, `hipaa`, `gdpr`; report section only ‚Äî evaluation always runs all rules) |
| `--fail-on {critical,high,medium,low,any,never}` | `high` | exit-code threshold (¬ß4) |
| `--include-passes` | off | passed-checks section (spec 005) |
| `--timestamp` | off | include `generated_at` in JSON output |
| `--quiet` / `-q` | off | suppress stderr progress/warnings summary |

Behavior order: validate PATH (must exist, be a directory) ‚Üí load rules (builtin unless
`--no-builtin`, plus `--rules`; zero rules total is an error) ‚Üí `parse_directory` ‚Üí
`evaluate` ‚Üí render ‚Üí write. Parser warnings (spec 002) print to **stderr** as
`warning W004 file.tf:12: message`, one line each, unless `--quiet`; the report itself goes
to stdout or `--output`. Never mix report bytes and diagnostics on the same stream.

### 3.2 `archscan graph PATH [--output FILE]`

Parse only; print the canonical graph JSON (spec 001). This is the debugging/inspection tool
("what did the parser actually see?") and, later, the input format other ingestors emit.

### 3.3 `archscan rules list [--rules PATH]... [--no-builtin] [--format {table,json}]`

List loaded rules: id, severity, category, title, frameworks mapped. `table` (default) is
aligned plain text; `json` is a sorted array. Also validates ‚Äî a broken custom rules dir
fails here with the full spec-003 error list, so users debug rules without running a scan.

### 3.4 Global

`archscan --version` prints `archscan <version>` (from `archassessor.__version__`).
`archscan --help` and per-command `--help` are accurate and include one usage example each.

## 4. Exit codes (the CI contract ‚Äî stable forever)

| Code | Meaning |
|------|---------|
| 0 | Ran successfully; no `fail`-verdict findings at/above `--fail-on` threshold |
| 1 | Ran successfully; threshold breached (`--fail-on never` makes 1 unreachable) |
| 2 | Usage error: bad flags, missing path, no rules loaded, unwritable output |
| 3 | Input error: rule files invalid (spec 003 `RuleLoadError`), or **all** `.tf` files unparseable (a *partially* parseable repo is not an error ‚Äî warnings + assessment of what parsed) |

`unknown` verdicts never affect the exit code in v1 (they're visible in the report; a
`--strict-unknown` flag is Phase 2 backlog). Uncaught exceptions are a bug by definition ‚Äî
`main()` wraps execution and converts expected error types to codes 2/3 with a one-line
stderr message (`error: ...`); anything else prints a bug-report request plus traceback.

## 5. UX details that matter

- Stderr summary line after a successful scan (unless `--quiet`):
  `scanned 24 resources, 28 rules ‚Üí score 70, findings: 1 critical, 2 high, 3 medium (see report)` .
- No color codes in v1 output (avoids TTY-detection complexity and keeps bytes deterministic;
  color is Phase 2 polish).
- All file I/O explicit UTF-8. Paths in messages shown relative to CWD when under it.

## 6. Acceptance criteria

End-to-end tests invoking `main(argv)` in-process (capsys) against the spec-002 fixtures:

1. `scan` on the `simple/` fixture with builtin rules ‚Üí exit 0 or 1 (fixture-dependent,
   pin it), stdout is valid Markdown starting `# Architecture Assessment Report`, stderr
   has the summary line.
2. `--format json -o out.json` writes valid JSON parseable back into an `Assessment`;
   stdout empty; deterministic across two runs (byte-equal files).
3. Exit-code matrix: fixture engineered to yield a `high` finding ‚Üí exit 1 with default
   threshold, 0 with `--fail-on critical`, 1 with `--fail-on any`, 0 with `--fail-on never`.
4. Nonexistent path ‚Üí exit 2, `error:` line on stderr, empty stdout.
5. `--rules` pointing at an invalid rule file ‚Üí exit 3, all load problems listed.
6. `--no-builtin` with no `--rules` ‚Üí exit 2 ("no rules loaded").
7. `graph` on `simple/` ‚Üí byte-identical to the fixture's expected-graph JSON.
8. `rules list` includes `BASE-SEC-001` in table output; `--format json` parses and is
   sorted by id.
9. `broken/` fixture (one bad file among good) ‚Üí scan still succeeds (exit 0/1), `W001`
   warning on stderr; with `--quiet`, stderr empty.
10. `--version` prints the package version; `--help` exits 0.
